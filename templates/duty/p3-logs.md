# Процедура P3 — проактивный анализ логов

## Зачем
1. Найти проблемы на тестовых стендах до того, как они попадут на ПРОМ.
2. Найти на ПРОМе неправильное поведение, которое не ловит мониторинг.
3. Улучшить сами логи — если разработчик читает их глазами, он видит, что доработать.

## Что ищем
Все ошибки в логах по модулям команды. Базовый инструмент — `logpilotskill`.

## Порядок работы

### 1. Подготовка
- Список модулей команды.
- Период анализа: с прошлого дежурства.
- Открыть Kibana / `logpilotskill`.

### 2. Обход стендов
Порядок: **сначала ПРОМ → потом тестовые стенды**.
По каждому стенду — все модули команды.

### 3. Анализ находки
Для каждой ошибки понять:
- Баг / шум / известная проблема / плохой лог.
- Разово или массово.

### 4. Действие
- **Баг** → тикет в бэклог команды-владельца.
- **Шум в логах** → тикет на чистку.
- **Плохой/неинформативный лог** → тикет на улучшение логирования.
- **Известная проблема** → упомянуть в отчёте, тикет не дублировать.

### 5. Запись в отчёт
Заполнить шаблон по каждому модулю. Если ничего не нашёл — зафиксировать явно.
